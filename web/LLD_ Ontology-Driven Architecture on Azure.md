# **Low-Level Design (LLD): Ontology-Driven Architecture with pgRDF, AGE, & Native JSONB**

This document outlines the Low-Level Design for a "Semantic Control Node" deployed as an Azure Container App (ACA). This architecture utilizes an **Ontology-Driven Architecture (ODA)**, where pgRDF acts as the proactive schema controller and validator for both Document (Native JSONB) and Graph (Apache AGE) representations, all backed by an Azure Database for PostgreSQL.

## **1\. Architectural Overview**

The system enforces a strict top-down schema hierarchy:

1. **Semantic Layer (pgRDF):** The Single Source of Truth. Business rules, taxonomies, and entity constraints are stored as RDF triples.  
2. **Proactive Controller (PL/pgSQL):** Event-driven functions that read the pgRDF ontology and automatically generate physical constraints for downstream models.  
3. **Operational Layers (AGE & JSONB):** \* **AGE:** Used for network traversal. Schemas (Vertex/Edge labels) are proactively generated by the Controller.  
   * **JSONB:** Used for high-speed document retrieval. Data integrity is enforced via CHECK constraints dynamically generated by the Controller.

### **Network Topology**

* **VNet Integration:** All components reside in a private Azure Virtual Network.  
* **Storage Layer:** Azure Database for PostgreSQL (Flexible Server).  
* **Compute Node:** Azure Container App (scales to zero) running the unified PostgreSQL image.

## **4\. Proactive Controller (Ontology-Driven DDL)**

This is the core of the ODA. These PL/pgSQL functions act as the "engine" that synchronizes the operational schemas with the ontological truth.

### **4.1. Synchronizing Ontology to JSONB Constraints**

This procedure extracts class requirements from pgRDF and builds physical CHECK constraints on the JSONB document table.

CREATE OR REPLACE PROCEDURE sync\_ontology\_to\_jsonb()  
LANGUAGE plpgsql  
AS $$  
DECLARE  
    rule RECORD;  
    constraint\_sql TEXT;  
BEGIN  
    \-- Step 1: Drop existing dynamic constraints to rebuild them  
    ALTER TABLE public.managed\_documents DROP CONSTRAINT IF EXISTS ontology\_jsonb\_check;

    \-- Step 2: Initialize the dynamic SQL builder  
    constraint\_sql := 'ALTER TABLE public.managed\_documents ADD CONSTRAINT ontology\_jsonb\_check CHECK (';  
      
    \-- Step 3: Query pgRDF for Class requirements  
    \-- Assuming a SPARQL query that returns ?className and ?requiredProperty  
    FOR rule IN   
        SELECT class\_name, required\_prop   
        FROM pgrdf\_query(  
            'SELECT ?class\_name ?required\_prop WHERE {   
                ?class rdfs:subClassOf sys:Document .  
                ?class sys:hasName ?class\_name .  
                ?class sys:requiresProperty ?required\_prop .  
            }', 'master\_ontology'  
        ) AS t(class\_name text, required\_prop text)  
    LOOP  
        \-- Step 4: Append to the CHECK constraint  
        \-- Logic: If the document is of this class, it MUST contain this property  
        constraint\_sql := constraint\_sql ||   
            format('(doc-\>\>''type'' \!= %L OR doc ? %L) AND ', rule.class\_name, rule.required\_prop);  
    END LOOP;

    \-- Step 5: Finalize and execute (appending 'TRUE' to close the trailing AND)  
    constraint\_sql := constraint\_sql || ' TRUE );';  
      
    EXECUTE constraint\_sql;  
    RAISE NOTICE 'JSONB Ontology Constraints Synchronized Successfully.';  
END;  
$$;

### **4.2. Synchronizing Ontology to Apache AGE Graph Schema**

This procedure ensures that all Nodes (Classes) and Edges (Predicates) defined in the ontology exist as Labels in the AGE Graph.

CREATE OR REPLACE PROCEDURE sync\_ontology\_to\_age()  
LANGUAGE plpgsql  
AS $$  
DECLARE  
    entity RECORD;  
BEGIN  
    \-- Ensure graph exists  
    PERFORM create\_graph('semantic\_graph') WHERE NOT EXISTS (  
        SELECT 1 FROM ag\_graph WHERE name \= 'semantic\_graph'  
    );

    \-- 1\. Sync Vertex Labels (Classes)  
    FOR entity IN   
        SELECT class\_name FROM pgrdf\_query(  
            'SELECT ?class\_name WHERE { ?class a owl:Class ; sys:hasName ?class\_name }', 'master\_ontology'  
        ) AS t(class\_name text)  
    LOOP  
        PERFORM create\_vlabel('semantic\_graph', entity.class\_name)   
        WHERE NOT EXISTS (SELECT 1 FROM ag\_label WHERE name \= entity.class\_name);  
    END LOOP;

    \-- 2\. Sync Edge Labels (Object Properties)  
    FOR entity IN   
        SELECT prop\_name FROM pgrdf\_query(  
            'SELECT ?prop\_name WHERE { ?prop a owl:ObjectProperty ; sys:hasName ?prop\_name }', 'master\_ontology'  
        ) AS t(prop\_name text)  
    LOOP  
        PERFORM create\_elabel('semantic\_graph', entity.prop\_name)  
        WHERE NOT EXISTS (SELECT 1 FROM ag\_label WHERE name \= entity.prop\_name);  
    END LOOP;

    RAISE NOTICE 'AGE Graph Schema Synchronized Successfully.';  
END;  
$$;

### **4.3. Event Trigger (The Automation)**

To make the controller proactive, a trigger fires whenever the ontology is updated, cascading the changes to the physical schemas.

CREATE OR REPLACE FUNCTION trigger\_ontology\_sync()  
RETURNS TRIGGER AS $$  
BEGIN  
    \-- Run the sync procedures asynchronously or immediately based on load  
    CALL sync\_ontology\_to\_jsonb();  
    CALL sync\_ontology\_to\_age();  
    RETURN NEW;  
END;  
$$ LANGUAGE plpgsql;

\-- Assuming 'ontology\_triples' is the underlying table for pgRDF data  
CREATE TRIGGER on\_ontology\_update  
AFTER INSERT OR UPDATE OR DELETE ON public.ontology\_triples  
FOR EACH STATEMENT EXECUTE FUNCTION trigger\_ontology\_sync();

## **5\. OntoSys Service Application (FastAPI)**

The application code no longer handles schema validation; it relies entirely on the database engine's proactive constraints.

from fastapi import FastAPI, HTTPException  
import psycopg2  
from contextlib import asynccontextmanager

PG\_CONFIG \= "host=pgrdf-node.internal.azurecontainerapps.io dbname=postgres user=postgres password=\*\*\*"

@asynccontextmanager  
async def lifespan(app: FastAPI):  
    \# Setup session variables globally for the connection pool  
    conn \= psycopg2.connect(PG\_CONFIG)  
    conn.autocommit \= True  
    with conn.cursor() as cur:  
        cur.execute("LOAD 'age';")  
        cur.execute("SET GLOBAL search\_path \= ag\_catalog, '$user', public;")  
    conn.close()  
    yield

app \= FastAPI(lifespan=lifespan)

@app.post("/api/v1/document")  
async def ingest\_document(doc: dict):  
    """  
    Attempts to insert a JSONB document.  
    If the document violates the pgRDF-generated CHECK constraint,   
    Postgres will reject it automatically.  
    """  
    conn \= psycopg2.connect(PG\_CONFIG)  
    try:  
        with conn.cursor() as cur:  
            import json  
            \# The database engine performs the validation transparently  
            cur.execute(  
                "INSERT INTO managed\_documents (doc) VALUES (%s) RETURNING id;",   
                (json.dumps(doc),)  
            )  
            doc\_id \= cur.fetchone()\[0\]  
            conn.commit()  
            return {"status": "success", "id": doc\_id}  
    except psycopg2.errors.CheckViolation as e:  
        conn.rollback()  
        raise HTTPException(status\_code=400, detail=f"Ontology Validation Failed: {e}")  
    finally:  
        conn.close()

## **6\. Infrastructure & Scaling Configuration**

### **Scale-to-Zero (KEDA TCP Scaling)**

Because the Controller runs inside the database, waking the database via a TCP connection guarantees the schema is ready before the application queries it.

* **Min Replicas:** 0  
* **Max Replicas:** 1 (Singleton Compute Node to avoid schema collision)  
* **Scale Rule:** tcp on port 5432
